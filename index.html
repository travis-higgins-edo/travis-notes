1. Background & The Core Question

EDO produces engagement lift metrics through two distinct pipelines:

  • AEC — primarily Vizio spine, 1:1 household-level exposure data, run through the CMPDS scoring model.
  • TV Ad Airings (Spike-based / Linear) — Google Trends-derived search engagement spikes attributable to individual TV ad airings.

In existing client reports, TV Ad Airings metrics are routinely compared across brands, while AEC metrics are kept within a brand or rely on specific brand-normalized index metrics. As we develop a chat agent on top of these data sources — one we cannot tightly control the way we control PDF reports — we need to understand why the comparison rules differ and what it would take to safely allow cross-brand AEC comparisons.

2. Methodology Differences: Why Cross-Brand Works in One But Not the Other

TV Ad Airings (Spike-based)

Primary metric: SEV (Search Engagement Volume)

SEV = added_traction / sev_impressions_factor

Normalization is baked in at the per-airing level via four factors computed across all airings in a normalization group:

  • duration_factor — ad length
  • entity_factor — brand baseline search volume
  • impressions_factor — audience reach
  • network_hour_factor — daypart / network

The entity_factor is the key — it divides out brand-level baseline differences before aggregation. Cross-brand SER comparisons are therefore valid at the data layer.

AEC (Vizio 1:1)

Primary metric: LIFT (Ad-Driven Engagement Rate)

LIFT = sum(lift × weight) / sum(weight)

Raw LIFT is not entity-normalized. It is the direct output of the CMPDS scoring model and inherently carries each brand's baseline scale (brand propensity, conversion probability, search volume).

Comparing Coca-Cola's LIFT to a niche brand's LIFT is comparing apples to oranges — each brand has a fundamentally different denominator.

Reports therefore derive an entity-normalized index after the fact:

brand_normalized_eri = lift / overall_brand_lift × 100

Each brand is re-based to a 100-point scale relative to its own overall performance.

Core insight: In TV Ad Airings, brand normalization is a continuous statistical adjustment applied at the per-airing level. In AEC, no equivalent per-exposure entity factor exists in the raw data — the reporting layer must construct an index after aggregation. This is why the rules for safe comparison differ.

3. Four Options for Cross-Brand AEC Comparison

Ranked from lowest effort / fastest to ship to highest effort / strongest parity with the TV Ad Airings model.

Option 1: Surface Existing Entity-Normalized Metrics as the Default Low Effort Ship First

The infrastructure already exists. brand_normalized_eri is computed in the VeraMax query layer and feeds the category-level indices used in existing reports.

What to do: In the chat agent, detect cross-brand intent and route to SUBJECT_AD_DRIVEN_ENGAGEMENT_RATE_ENTITY_NORMALIZED_INDEX (and its category / display-index variants) instead of raw LIFT. Always caveat: "100 = each brand's own average; values above 100 mean above-average performance for that brand."

Limitation: Compares relative patterns, not absolute magnitudes. You can say "Brand A over-indexes on Hulu more than Brand B does" but not "Brand A drives more engagement than Brand B."

Option 2: Cross-Brand Absolute Engagement Rate Index Moderate Effort

Build a new metric that directly answers "Which brand drives more engagement?":

Brand Absolute Index = brand_overall_lift / weighted_avg(all_brands_overall_lift_in_compset) × 100

Scaffolding for this already exists in the metrics spec under SUBJECT_AD_DRIVEN_BRAND_ENGAGEMENT_RATE and SUBJECT_AD_DRIVEN_BRAND_ENGAGEMENT_RATE_INDEX.

What to do: Add one CTE to the VeraMax query chain that computes the weighted average of overall_brand_lift across the competitive set, then divides each brand's value by it.

Advantage: Direct cross-brand comparable index, conceptually parallel to how entity_factor works in TV Ad Airings. Combined with Option 1, you can produce a dimension-specific cross-brand index by multiplication.

Option 3: Compute an AEC-Native Entity Factor in CMPDS High Effort

Mirror the TV Ad Airings architecture by exporting an entity-normalized lift column from the CMPDS pipeline into the RRD tables:

ENTITY_NORMALIZED_LIFT = NEW_LIFT / overall_brand_lift

The pipeline already computes BRAND_MULT and LIFT_RATIO internally for model calibration; this would expose an analogous divisible factor in the output schema.

Advantage: True data-layer parity with TV Ad Airings. The chat agent and any downstream consumer can compare across brands without special routing logic.

Cost: Requires CMPDS pipeline changes, RRD schema migration, and downstream query updates. Long-term ideal, not a quick win.

Option 4: Chat Agent Guardrails (Complementary to Any Above) Low Effort Required

Regardless of metric strategy, the chat agent needs a guardrail layer that:

  1. Detects comparison intent — "compare X vs Y" routes to a cross-brand pathway.
  2. Auto-selects the right metric — raw LIFT for within-brand questions, entity-normalized index for cross-brand.
  3. Always caveats — surfaces the competitive set, the normalization approach, and what "100" means.
  4. Refuses unsafe comparisons — if brands are not in the same competitive set / scoring run, refuse rather than produce a misleading number.

4. The Cross-Dimension Problem

Cross-brand comparison is one problem. Cross-dimension comparison is a separate, second problem that none of the metric options above directly solve.

Worked Example

User asks: Compare GEICO on streaming/Amazon vs. ALLSTATE on streaming/Amazon/Monday.

GEICO has 2 dimensional filters applied. ALLSTATE has 3. You are comparing a broader slice of GEICO data against a narrower slice of ALLSTATE data. The denominators of any normalized index will be different, and the comparison is structurally invalid — like comparing a weekly batting average to a monthly one.

Why the Query Architecture Cannot Mix Dimensions

The VeraMax query layer parameterizes every query by a single group_columns array (e.g., ['BRAND', 'PLATFORM', 'PUBLISHER_1P']) applied uniformly to all brands in the competitive set. There is no mechanism to ask for one brand at one dimensional cut and another brand at a different cut in the same query.

In TV Ad Airings this problem barely surfaces because each airing is an atomic unit — one airing, one brand, one network, one time — and normalization factors apply at that atomic level. In AEC, the RRD tables are pre-aggregated by dimension group, so two rows with different group_columns arrays represent fundamentally different aggregation scopes.

5. Three Tiers of Comparison Validity

This is the framework the chat agent should reason about for every comparison request:

1
Same brand, same dimensions
Example: GEICO streaming/Amazon vs. GEICO streaming/Hulu
Metric: Raw LIFT or Engagement Rate Index
No normalization needed.
Valid
2
Cross-brand, same dimensions
Example: GEICO streaming/Amazon vs. ALLSTATE streaming/Amazon
Metric: Entity-normalized index (Option 1) or Cross-Brand Absolute Index (Option 2)
Comparison valid with normalization.
Valid w/ Norm
3
Cross-brand, different dimensions
Example: GEICO streaming/Amazon vs. ALLSTATE streaming/Amazon/Monday
Metric: No single metric saves this.
Must align dimensions or decompose into independent statements.
Invalid

6. Chat Agent Design Recommendations

Decision Flow

User asks a cross-brand comparison question
↓
Do both brands use the same dimensional cut?
Yes Use entity-normalized index (Option 1 or Option 2)
Direct comparison is valid. Always caveat: 100 = each brand's own average.
No Mismatched dimensions — do not produce a direct comparison
Choose one of two paths:
Path A · Preferred Align dimensions
Ask the user which cut to use, then re-run both brands at the same dimensions. "I'll compare both brands at the same cut. Should I use the broader cut [streaming · Amazon], or apply the Monday filter to GEICO as well?"
Path B · Fallback Decompose into independent statements
Present each brand's result on its own terms with an explicit non-comparability disclaimer. GEICO on streaming/Amazon: +15% vs. its campaign average.
ALLSTATE on streaming/Amazon/Monday: −8% vs. its campaign average.
Note: these use different dimensional cuts and are not directly comparable.

Required Behaviors

  • Intent detection: Classify every metric request as within-brand, cross-brand-aligned, or cross-brand-mismatched.
  • Dimension alignment check: Before producing a cross-brand number, verify both sides use the same group_columns. If not, prompt to align.
  • Mandatory caveats: Cross-brand outputs always include the competitive set definition, the normalization basis, and the meaning of "100".
  • Refusal mode: When brands are in different scoring runs, different competitive sets, or different time windows, refuse the comparison rather than produce a misleading number.
  • Telemetry: Log every comparison with its tier classification so we can audit how the agent handles edge cases in production.

7. Summary & Recommended Next Steps

What We're Solving

Move from tightly-controlled PDF reports (where brand normalization was implicit in report design) to a chat agent that produces correct, comparable answers without human pre-flight checks.

Recommended Path

  1. Ship Option 1 + Option 4 immediately. Use the existing entity-normalized indices and wrap them with chat agent guardrails. This unblocks Tier-2 comparisons today with no data layer changes.
  2. Build Option 2 next. Add a Cross-Brand Absolute Index CTE to VeraMax. This gives clients the "which brand wins" answer they actually want.
  3. Plan Option 3 as the long-term ideal. An AEC-native entity factor would give true data-layer parity with TV Ad Airings, but requires CMPDS pipeline and RRD schema work.
  4. Treat the cross-dimension problem as a query-alignment problem, not a metric problem. No new metric solves Tier 3; the chat agent must enforce dimensional alignment or decompose the answer.
Bottom line: Options 1 and 2 unblock cross-brand comparisons at the same dimensional cut. For mismatched dimensions, the answer is not a new metric — it is query alignment enforced by the chat agent. Option 4 (guardrails) is non-negotiable regardless of which metric path we choose.
